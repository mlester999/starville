# Phase 11D progression trust boundaries

The browser submits intent and expected revisions only. PostgreSQL owns event eligibility, XP
amounts, curve lookup, level calculation, unlock requirements, objective progress, achievement
criteria, reward settlement, title ownership, and public profile projection.

Direct browser access to progression tables and RPCs is revoked. The API authorizes the player
wallet or administrator, checks trusted mutation origins, parses strict contracts, disables response
caching, and calls narrow service-role RPCs. Database functions use an empty `search_path`, row or
advisory locks, bounded inputs, durable rate limits, and append-only evidence. RLS is enabled and
forced on private progression state.

Exact-once protection combines canonical source UUIDs, a unique player/event/source/skill index,
idempotency keys, locked progress rows, immutable rule versions, and original-result replay.
Preview, failed, duplicate, foreign-owner, and Game Test actions are not authoritative sources.
Suspension and rename-required state block player progression mutations.

Administrator changes require permission checks inside the database, verified sessions, AAL2 where
sensitive, optimistic revisions, reasons, bounded rate limits, and append-only audits. Corrections
are compensating events. Reconciliation cannot invent XP. Live-ops pauses do not erase history.

Owner event payloads contain only progression-safe data. Public profile projection contains level,
equipped title key, and selected badge key—not wallet, inventory, DUST, quest detail, XP history,
moderation, or session data. Structured errors avoid revealing another owner’s record.

No Phase 11D function can sign a wallet transaction, issue STAR, mint an NFT, convert DUST, sell XP,
or enable a token claim.
